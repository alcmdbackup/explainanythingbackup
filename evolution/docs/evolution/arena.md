# Arena

Persistent cross-method comparison system using OpenSkill (Weng-Lin Bayesian) ratings. The Arena provides a unified rating pool — pipeline variants are loaded from the Arena at run start and synced back at completion, so in-run and cross-run ratings share the same scale.

## Overview

The Arena stores articles organized by **topics** (unique prompts). Each topic contains **entries** — articles generated by different methods/models. Entries are ranked via OpenSkill ratings computed from head-to-head LLM-judged comparisons.

**Unified pool model:** At pipeline start, existing Arena entries are loaded into `state.pool` with pre-seeded ratings. The CalibrationRanker and Tournament compare them naturally alongside new variants. At finalization, all new variants, matches, and updated ratings are synced back atomically via the `sync_to_arena` RPC.

### Key workflows:
- **Workflow A** (1-shot): Generate an article from a prompt using any LLM model, add to Arena
- **Workflow B** (evolution): Pipeline variants are automatically synced to Arena at run completion
- **Workflow C** (comparison): Run Swiss-style pairwise comparisons between entries in a topic

## Key Concepts

### OpenSkill Rating System

The Arena uses OpenSkill (Weng-Lin Bayesian) ratings — the same algorithm used within pipeline runs (see [Rating & Comparison](./rating_and_comparison.md)):

- Each entry has `{mu, sigma}`: `mu` is estimated skill, `sigma` is uncertainty. New entries start at `mu=25, sigma=8.333`.
- **Ordinal** = `mu - 3*sigma` — conservative estimate that penalizes uncertain entries.
- **`elo_rating`** is a derived display column mapping ordinal to a 0–3000 scale. Used for sort order only.
- **`display_elo`** = `ordinalToEloScale(mu)` — the point estimate shown in the leaderboard UI. Always inside the 95% CI bounds (`mu ± 1.96*sigma`), unlike `elo_rating` which uses ordinal (`mu - 3*sigma`) and can fall outside CI.
- **`elo_per_dollar`** — cost-efficiency metric.
- **Calibrated entries** (sigma < 5.0) skip re-calibration when loaded into pipeline runs but still serve as opponents.

### Unified Pool Loading

At pipeline start, `loadArenaEntries()`:
1. Resolves the topic from `prompt_id` or explanation title
2. Queries all active entries + elo data from the Arena
3. Pushes entries into `state.pool` with `fromArena: true`
4. Pre-seeds `state.ratings` and `state.matchCounts`

These entries participate in all ranking naturally but are filtered out of persistence (variants table), attribution, agent metrics, run summary, and flow critiques.

### Arena Sync

At pipeline finalization, `syncToArena()`:
1. Collects all new variants (`!fromArena`) with their ratings
2. Collects all match history from the run
3. Collects updated elo for both new and existing Arena entries
4. Calls `sync_to_arena` RPC for atomic upsert

## Architecture

### Database Schema

Four tables (prefixed `evolution_arena_`):

| Table | Purpose |
|-------|---------|
| `evolution_arena_topics` | Prompt-based grouping with case-insensitive unique index |
| `evolution_arena_entries` | Articles with FKs to evolution runs/variants, `generation_method` CHECK |
| `evolution_arena_comparisons` | Match history with confidence scores and judge model |
| `evolution_arena_elo` | Per-topic OpenSkill ratings (mu, sigma, ordinal) with derived display columns |

### Generation Methods

| Method | Description |
|--------|-------------|
| `oneshot` | Direct LLM generation from prompt |
| `evolution` | Pipeline-generated variant synced to Arena |
| `evolution_winner` | Legacy: explicit winner from pipeline run |
| `evolution_baseline` | Legacy: seed article from pipeline run |
| `evolution_ranked` | Legacy: ranked entry from pipeline run |
| `evolution_top3` | Legacy: kept for backward compatibility |

### Server Actions

All actions in `evolution/src/services/arenaActions.ts` follow the `ActionResult<T>` + `requireAdmin()` + `withLogging` pattern.

### CLI Scripts

| Script | Purpose |
|--------|---------|
| `evolution/scripts/add-to-arena.ts` | Add evolution run winner to Arena |
| `evolution/scripts/run-arena-comparison.ts` | CLI for pairwise comparisons |
| `evolution/scripts/run-prompt-bank.ts` | Batch generation orchestrator |
| `evolution/scripts/run-prompt-bank-comparisons.ts` | Batch comparison runner |

### Topic Archiving

Topics can be archived via the Admin UI (topic list or topic detail page). Archived topics are:
- **Hidden** from the Arena topic list and cross-topic summary by default (toggle "Show archived" to reveal)
- **Hidden** from experiment prompt selection (`getPromptsAction` filters by `status: 'active'`)
- **Still accessible** for existing pipeline runs — `loadArenaEntries`, `syncToArena`, and `resolveTopicId` do not filter by status, so in-progress or future runs referencing an archived topic continue to work

Archive/unarchive uses the `status` column on `evolution_arena_topics` (`'active'` | `'archived'`), distinct from soft-delete (`deleted_at`).

### Admin UI

Two pages under `/admin/evolution/arena/`:

- **Topic List** (`page.tsx`): Cross-topic efficiency summary, prompt bank coverage grid, topics table with archive/unarchive and "Show archived" toggle
- **Topic Detail** (`[topicId]/page.tsx`): Leaderboard (shows `display_elo` rating, `run_cost_usd` from linked evolution run, strategy label, experiment name), Cost vs Rating scatter (uses `display_elo`, enhanced tooltip with method+model, quadrant subtitle), Match History, Text Diff, archive/unarchive button, **budget tier filter** dropdown (All / ≤$0.25 / $0.25–$0.50 / $0.50–$1.00) filtering entries by `run_budget_cap_usd` on `ArenaEloEntry`
