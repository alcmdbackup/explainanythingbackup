# Arena

Persistent cross-method comparison system using OpenSkill (Weng-Lin Bayesian) ratings. The Arena provides a unified rating pool тАФ pipeline variants are loaded from the Arena at run start and synced back at completion, so in-run and cross-run ratings share the same scale.

## Overview

The Arena stores articles organized by **topics** (unique prompts). Each topic contains **variants** тАФ articles generated by different methods/models, marked with `synced_to_arena = true` on `evolution_variants`. Variants are ranked via OpenSkill ratings computed from head-to-head LLM-judged comparisons.

**Unified pool model:** At pipeline start, existing Arena entries are loaded into the pool with pre-seeded ratings. The `rankPool()` operation compares them naturally alongside new variants (triage for new entrants, then Swiss fine-ranking among contenders). At finalization, all new variants, matches, and updated ratings are synced back atomically via the `sync_to_arena` RPC.

### Key workflows:
- **Workflow A** (1-shot): Generate an article from a prompt using any LLM model, add to Arena
- **Workflow B** (evolution): Pipeline variants are automatically synced to Arena at run completion
- **Workflow C** (comparison): Run Swiss-style pairwise comparisons between entries in a topic

## Key Concepts

### OpenSkill Rating System

The Arena uses OpenSkill (Weng-Lin Bayesian) ratings тАФ the same algorithm used within pipeline runs (see [Rating & Comparison](./rating_and_comparison.md)):

- Each variant has `{mu, sigma}`: `mu` is estimated skill, `sigma` is uncertainty. New entries start at `mu=25, sigma=8.333`.
- **Ranking** uses `mu` directly тАФ the leaderboard sorts by `mu DESC`.
- **`elo_score`** = `toEloScale(mu)` тАФ derived display column mapping mu to a 0тАУ3000 scale via `1200 + (mu - 25) * 16`. A fresh entry (mu=25) shows as Elo 1200.
- **`elo_per_dollar`** тАФ cost-efficiency metric based on `toEloScale(mu)`.
- **Calibrated entries** (sigma < 5.0) skip re-calibration when loaded into pipeline runs but still serve as opponents.

### Unified Pool Loading

At pipeline start, `loadArenaEntries()`:
1. Resolves the topic from `prompt_id` or explanation title
2. Queries all active arena variants (`synced_to_arena = true`) with their ratings from `evolution_variants`
3. Pushes entries into `state.pool` with `fromArena: true`
4. Pre-seeds `state.ratings` and `state.matchCounts`

These entries participate in all ranking naturally but are filtered out of persistence (variants table), attribution, agent metrics, run summary, and flow critiques.

### Arena Sync

At pipeline finalization, `syncToArena()`:
1. Collects all new variants (`!fromArena`) with their ratings
2. Collects all match history from the run
3. Collects updated ratings for both new and existing arena variants
4. Calls `sync_to_arena` RPC тАФ INSERT ON CONFLICT targets `evolution_variants`, setting `synced_to_arena = true` and populating arena columns (mu, sigma, prompt_id, generation_method, model, cost_usd, arena_match_count)

## Architecture

### Database Schema

Two tables:

| Table | Purpose |
|-------|---------|
| `evolution_variants` | Articles with arena columns (mu, sigma, prompt_id, synced_to_arena, generation_method, model, cost_usd, arena_match_count, archived_at, evolution_explanation_id). Arena variants identified by `synced_to_arena = true`. |
| `evolution_arena_comparisons` | Match history with confidence scores and judge model |

Prompt-based grouping is handled by `evolution_prompts` (shared with the rest of the pipeline, not arena-specific).

### Generation Methods

| Method | Description |
|--------|-------------|
| `oneshot` | Direct LLM generation from prompt |
| `pipeline` | V2 pipeline-generated variant synced to Arena |
| `evolution_winner` | Legacy: explicit winner from pipeline run |
| `evolution_baseline` | Legacy: seed article from pipeline run |
| `evolution_ranked` | Legacy: ranked entry from pipeline run |
| `evolution_top3` | Legacy: kept for backward compatibility |

### Server Actions

All actions in `evolution/src/services/arenaActions.ts` follow the `ActionResult<T>` + `requireAdmin()` + `withLogging` pattern.

### CLI Scripts

| Script | Purpose |
|--------|---------|
| `evolution/scripts/add-to-arena.ts` | Add evolution run winner to Arena |
| `evolution/scripts/run-arena-comparison.ts` | CLI for pairwise comparisons |
| `evolution/scripts/run-prompt-bank.ts` | Batch generation orchestrator |
| `evolution/scripts/run-prompt-bank-comparisons.ts` | Batch comparison runner |

### Topic Archiving

Topics can be archived via the Admin UI (topic list or topic detail page). Archived topics are:
- **Hidden** from the Arena topic list and cross-topic summary by default (toggle "Show archived" to reveal)
- **Hidden** from experiment prompt selection (`getPromptsAction` filters by `status: 'active'`)
- **Still accessible** for existing pipeline runs тАФ `loadArenaEntries`, `syncToArena`, and `resolveTopicId` do not filter by status, so in-progress or future runs referencing an archived topic continue to work

Archive/unarchive uses the `status` column on `evolution_prompts` (`'active'` | `'archived'`), distinct from soft-delete (`deleted_at`).

### Admin UI

The V1 arena admin pages have been removed. Arena data is managed via CLI scripts in `evolution/scripts/deferred/`.
