## Scope (refined with user, 2026-06-05/06)
The tool is a **"prompt playground"** — *test the prompt*, not the agent. It operates on two **rewrite units**, each a **single prompt → single LLM call → single raw output**:

1. **Whole-article rewrite** — an editable prompt (a generate-tactic preset, or fully custom) + a source article → one generation call → the rewritten article. *No ranking, no pool, no Elo, no scoring.*
2. **Paragraph rewrite** — an editable prompt (a rewrite-directive preset, or fully custom) + a source paragraph → one rewrite call → the rewritten paragraph. *No per-slot ranking, no recombine.*

A **run** = **one shared source input** + **N config cards**, each independently setting `{ label, prompt, model, temperature }`. "Run all" dispatches every config as an independent single LLM call in parallel; raw outputs render side-by-side with per-config cost. Shared input makes side-by-side apples-to-apples (and makes the future judging phase meaningful).

**Explicitly OUT of v1 scope → Future Scope** (see section below): recombine, iterative editing, criteria agents, debate, any chained/multi-call rewrite; relative judging / Elo ranking of a pool; editing the judge prompt; matrix auto-expansion of configs; persisted/shareable sessions.

## Problem
Researchers iterating on the evolution rewrite prompts have no fast feedback loop: to see what a tweaked prompt/temperature/model produces they must author a strategy, enqueue a run, wait for the full generate→rank→evolve pipeline, then dig through invocation detail. There is no way to A/B several `{prompt, temperature, model}` configurations against the same source text in one view. This makes prompt engineering slow and indirect. The playground provides an on-demand, side-by-side, single-call rewrite harness that surfaces raw outputs + cost. It writes **no evolution-pipeline rows** (no run/invocation/variant/metric/arena). Like every app LLM call, each rewrite still records one `llmCallTracking` row (`call_source='evolution_playground'`) and draws on the shared daily `evolution` spend budget — this is `callLLM`'s built-in behaviour and is desirable for cost auditing (see Design Decisions).

## Options Considered
- [x] **Option A: Single-call playground — CHOSEN (v1)**: an API route + a thin harness that, per config, builds the rewrite prompt (reusing the pipeline's pure prompt builders), makes **one `callLLM`**, reads cost from the `onUsage` callback, runs display-only format validation, and returns the output. No agent orchestration, **no evolution-pipeline rows**, no migrations. (Each `callLLM` still writes one `llmCallTracking` row + consumes the shared `evolution` budget — unavoidable and desirable for cost auditing.) Outputs live only in the browser session.
- [ ] **Option B: Reuse `Agent.execute()`**: call `GenerateFromPreviousArticleAgent.execute()` / `ParagraphRecombineAgent.execute()`. Rejected for v1 — those bundle ranking (GFPA) and slot fan-out + recombine (ParagraphRecombine), which is exactly the chained complexity we are deferring. The single-call path is simpler, cheaper, faster, and has no DB-pollution surface at all.
- [ ] **Option C: Persisted playground sessions / 1-iteration runs**: tables or run rows for history. Rejected for v1 (migrations, RLS, analytics pollution). Revisit in Future Scope.

## Design Decisions (confirmed with user; corrected against codebase in plan-review iter 1)
- **Architecture: single `callLLM` per config.** NO `Agent.execute()`, NO `AgentContext`, NO `V2CostTracker`/`AgentCostScope`, NO pool/ratings/cache, NO recombine/ranking. Harness = `buildPrompt → callLLM(one call, capture usage via onUsage) → validate(display-only)`.
- **`callLLM` contract (VERIFIED `src/lib/services/llms.ts`):** signature is `callLLM(prompt, call_source, userId, model, streaming=false, setText?, responseObj?, responseObjName?, debug=false, options?)` and **returns `Promise<string>`** (text only). `model` is the **4th positional arg** (not in options); `temperature` lives in `options`. **Pass `null` (NOT `undefined`) for `setText`/`responseObj`/`responseObjName`** — `validateStreamingArgs` throws if `setText !== null` when `streaming=false`; the pipeline passes `null, null, null` (`claimAndExecuteRun.ts:200`). **Token usage + cost arrive via `options.onUsage(usage: LLMUsageMetadata)`** — `usage.estimatedCostUsd` IS the per-call cost (no separate cost call needed). Mirror the capture pattern in `evolution/src/lib/pipeline/claimAndExecuteRun.ts:218-233`. `userId = ANONYMOUS_USER_UUID` (exported from `llms.ts`; intentional divergence from the pipeline's `EVOLUTION_SYSTEM_USERID` — fine for playground attribution). `call_source='evolution_playground'` (the `evolution_` prefix routes spend to the shared daily `evolution` budget category + engages the LLM semaphore).
- **NOT zero-write (corrected):** `callLLM` ALWAYS writes one `llmCallTracking` row and reserves/reconciles budget via `LLMSpendingGate` — there is no suppress flag. The playground writes **no** `evolution_agent_invocations`/`evolution_variants`/`evolution_metrics`/`evolution_arena_comparisons` rows. The integration test asserts BOTH (zero evolution_* rows AND ≥1 `llmCallTracking` row with `call_source='evolution_playground'`).
- **Two units:** whole-article (`buildEvolutionPrompt`, FORMAT_RULES auto-appended) and paragraph (`buildParagraphRewritePrompt` scaffolding around an editable directive).
- **Custom prompts:** every config's prompt is fully editable. Article = editable `preamble` + `instructions` (FORMAT_RULES auto-wrapped + shown). Paragraph = editable directive (length/format scaffolding auto-wrapped + shown). Presets (24 tactics / 3 directives) are editable starting points; "(custom)" once edited. (Fully-raw whole-prompt override = Future Scope.)
- **Shared input per run:** one source field; configs vary only `{prompt, model, temperature}`. New input = new run.
- **Parallel:** `Promise.allSettled`; one config's failure is isolated. Cap: default **5** cards, hard max **10**.
- **Transport (SINGLE path — corrected):** the page **POSTs to the API route** `src/app/api/evolution/playground/route.ts` (`export const maxDuration = 300`, inline `requireAdmin()` + Zod + harness), mirroring `src/app/api/evolution/run/route.ts`. **No separate `'use server'` adminAction** — `maxDuration` applies only to route segments, not to a server action invoked from a page (this repo never sets it on a page segment), so a bare action would NOT get the 300s headroom. Host-gating is automatic: `src/middleware.ts` 404s BOTH `/api/evolution/*` and `/admin/evolution/*` on the public host (no middleware change).
- **Model selection (allowlisted):** the dropdown lists `getEvolutionModelIds()` (`src/config/modelRegistry.ts`) — the same enum `callLLM` validates against (off-list ⇒ ZodError). Playground spend shares the daily `evolution` cap with real runs (documented in `cost_optimization.md`).
- **Temperature:** `getModelMaxTemperature(model)` returns `number | null | undefined`. `null` **or `undefined`** ⇒ omit temperature (input disabled/hidden); a number ⇒ clamp `Math.min(userTemp, maxTemp)`.
- **Error taxonomy (real — `LLMRefusalError` is NOT thrown by `callLLM`):** `GlobalBudgetExceededError` ⇒ `budget`; `LLMKillSwitchError` ⇒ `killed`; provider/abort timeout ⇒ `timeout`; `ZodError`/other ⇒ `error`. A model **refusal returns as ordinary text** ⇒ `status:'success'` (refusal text shown; optional non-blocking `looksLikeRefusal` display heuristic). No `refused` status.
- **Validation is display-only:** `validateFormat` / `validateParagraphRewrite` shown as an informational chip (e.g. `would-drop: no_bullets`); **never blocks** output.
- **Spend guardrails:** the global `LLMSpendingGate` is the hard backstop, but it reserves a flat estimate (≈1000/4096 tokens) that under-counts long articles — so the **primary** guard is a per-run pre-flight estimate: Σ over configs of `calculateLLMCost(model, prompt.length/4, cappedOutputTokens)` (`@/config/llmPricing` — same helper `callLLM` uses); reject the run if total > **$0.50** (hardcoded v1). Restricting the model list keeps the worst case bounded.
- **Results layout:** responsive grid (wraps 2–3 per row). Manual cards only (no matrix expand in v1).
- **Kill-switch / rollback:** the route + nav item are gated behind env `EVOLUTION_PLAYGROUND_ENABLED` (default `'1'`); set `'0'` ⇒ route returns 403 and the nav item hides. Rollback = flip the flag or revert the PR (no migration).
- **Discovery:** linked from the evolution sidebar **Overview** group + a card on `/admin/evolution-dashboard`.

## Data Shapes (harness contract)
```ts
type RewriteUnit = 'article' | 'paragraph';
interface PlaygroundConfig { label: string; prompt: PromptSpec; model: string; temperature?: number; }
//   PromptSpec (article)   = { preamble: string; instructions: string }
//   PromptSpec (paragraph) = { directive: string }
interface PlaygroundRunInput { unit: RewriteUnit; sourceText: string; configs: PlaygroundConfig[]; }
interface PlaygroundConfigResult {
  label: string; output: string | null; costUsd: number;       // costUsd = usage.estimatedCostUsd from onUsage
  model: string; temperatureUsed: number | null; durationMs: number;
  status: 'success' | 'budget' | 'killed' | 'timeout' | 'error'; // refusals are 'success' (text shown)
  formatValid: boolean; formatIssues?: string[]; looksLikeRefusal?: boolean; errorMsg?: string;
}
interface PlaygroundRunResult { configs: PlaygroundConfigResult[]; totalCostUsd: number; }
```
