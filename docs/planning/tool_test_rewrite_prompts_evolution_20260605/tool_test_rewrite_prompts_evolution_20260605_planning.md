## Scope (refined with user, 2026-06-05/06)
The tool is a **"prompt playground"** — *test the prompt*, not the agent. It operates on two **rewrite units**, each a **single prompt → single LLM call → single raw output**:

1. **Whole-article rewrite** — an editable prompt (a generate-tactic preset, or fully custom) + a source article → one generation call → the rewritten article. *No ranking, no pool, no Elo, no scoring.*
2. **Paragraph rewrite** — an editable prompt (a rewrite-directive preset, or fully custom) + a source paragraph → one rewrite call → the rewritten paragraph. *No per-slot ranking, no recombine.*

A **run** = **one shared source input** + **N config cards**, each independently setting `{ label, prompt, model, temperature }`. "Run all" dispatches every config as an independent single LLM call in parallel; raw outputs render side-by-side with per-config cost. Shared input makes side-by-side apples-to-apples (and makes the future judging phase meaningful).

**Explicitly OUT of v1 scope → Future Scope** (see section below): recombine, iterative editing, criteria agents, debate, any chained/multi-call rewrite; relative judging / Elo ranking of a pool; editing the judge prompt; matrix auto-expansion of configs; persisted/shareable sessions.

## Problem
Researchers iterating on the evolution rewrite prompts have no fast feedback loop: to see what a tweaked prompt/temperature/model produces they must author a strategy, enqueue a run, wait for the full generate→rank→evolve pipeline, then dig through invocation detail. There is no way to A/B several `{prompt, temperature, model}` configurations against the same source text in one view. This makes prompt engineering slow and indirect. The playground provides an on-demand, side-by-side, single-call rewrite harness that surfaces raw outputs + cost, with zero DB/analytics side effects.

## Options Considered
- [x] **Option A: Ephemeral single-call playground — CHOSEN (v1)**: an API route + a thin harness that, per config, builds the rewrite prompt (reusing the pipeline's pure prompt builders), makes **one raw LLM call**, computes cost from token usage, runs display-only format validation, and returns the output. No agent orchestration, no DB writes, no migrations. Outputs live only in the browser session.
- [ ] **Option B: Reuse `Agent.execute()`**: call `GenerateFromPreviousArticleAgent.execute()` / `ParagraphRecombineAgent.execute()`. Rejected for v1 — those bundle ranking (GFPA) and slot fan-out + recombine (ParagraphRecombine), which is exactly the chained complexity we are deferring. The single-call path is simpler, cheaper, faster, and has no DB-pollution surface at all.
- [ ] **Option C: Persisted playground sessions / 1-iteration runs**: tables or run rows for history. Rejected for v1 (migrations, RLS, analytics pollution). Revisit in Future Scope.

## Design Decisions (confirmed with user)
- **Architecture: single LLM call per config.** NO `Agent.execute()`, NO `AgentContext`, NO `V2CostTracker`/`AgentCostScope`, NO pool/ratings/cache, NO recombine/ranking. Harness = `buildPrompt → callLLM(one call) → calculateLLMCost(usage) → validate(display-only)`.
- **Two units:** whole-article (`buildEvolutionPrompt`, FORMAT_RULES auto-appended) and paragraph (`buildParagraphRewritePrompt` scaffolding around an editable directive).
- **Custom prompts:** every config's prompt is fully editable. Article = editable `preamble` + `instructions` (anything; FORMAT_RULES auto-wrapped + shown). Paragraph = editable directive (length/format scaffolding auto-wrapped + shown). Presets (24 tactics / 3 directives) are editable starting points; "(custom)" once edited. (Fully-raw whole-prompt override = Future Scope.)
- **Shared input per run:** one source field; configs vary only `{prompt, model, temperature}`. New input = new run.
- **Parallel:** `Promise.allSettled`; one config's failure (refusal/timeout/budget) is isolated. Cap: default **5** cards visible, hard max **10**.
- **Transport:** API route `src/app/api/evolution/playground/route.ts` with `export const maxDuration = 300`.
- **Temperature:** disable the input when `getModelMaxTemperature(model) === null` (e.g. o3-mini); else clamp `Math.min(userTemp, maxTemp)`.
- **Validation is display-only:** `validateFormat` (article) / `validateParagraphRewrite` (paragraph) shown as an informational chip (e.g. `would-drop: no_bullets`); **never blocks** output.
- **Spend guardrails:** global `LLMSpendingGate` (hard backstop at `callLLM`) + a per-run pre-flight cost-cap estimate (reject the run if estimated total exceeds the cap). Cap default **$0.50/run**, hardcoded for v1.
- **Results layout:** responsive grid (wraps 2–3 per row as count grows). Manual cards only (no matrix expand in v1).
- **Ephemeral:** no DB writes anywhere (no agent machinery touches the DB).
- **Discovery:** linked from the evolution sidebar **Overview** group + a card on `/admin/evolution-dashboard`.

## Data Shapes (harness contract)
```ts
type RewriteUnit = 'article' | 'paragraph';
interface PlaygroundConfig { label: string; prompt: PromptSpec; model: string; temperature?: number; }
//   PromptSpec (article)   = { preamble: string; instructions: string }
//   PromptSpec (paragraph) = { directive: string }
interface PlaygroundRunInput { unit: RewriteUnit; sourceText: string; configs: PlaygroundConfig[]; }
interface PlaygroundConfigResult {
  label: string; output: string | null; costUsd: number; model: string; temperatureUsed: number | null;
  durationMs: number; status: 'success' | 'refused' | 'budget' | 'timeout' | 'error';
  formatValid: boolean; formatIssues?: string[]; errorMsg?: string;
}
interface PlaygroundRunResult { configs: PlaygroundConfigResult[]; totalCostUsd: number; }
```

## Future Scope (deliberately deferred — design v1 to make these cheap)
- **Relative judging / Elo of a pool** (fast-follow): a `⚖ Judge this pool` button over the N outputs (which are already a first-class pool sharing one input). Round-robin (cap ~6) + 2-pass reversal reusing `buildComparisonPrompt`/`compareWithBiasMitigation`/`parseWinner`/`createRating`/`updateRating`/`updateDraw`/`ComparisonCache`; in-memory Elo ± uncertainty, framed directional; ephemeral (no arena writes). Requires identical source input within the judged group (already true).
- **Judge-prompt editing** — handled by a separate parallel project; integrate later (do not build here).
- **Matrix expand** — generate cards from M prompts × K models.
- **Additional rewrite families** — criteria agents, iterative editing, debate, full paragraph-recombine.
- **Fully-raw whole-prompt override**, **persisted/shareable sessions**, **researcher-adjustable cost cap**.
