# Comparison Infrastructure Plan

## Background
The evolution pipeline generates and refines article variants using cheap LLM models (deepseek-chat, gpt-4.1-nano) through iterative Elo-ranked competition. However, there is no way to evaluate whether this iterative approach produces better results than simply calling an expensive model once. The existing infrastructure lacks three capabilities: generating articles from prompts (rather than existing articles), 1-shot generation with expensive models, and a persistent cross-run comparison system (an "article bank") with its own Elo ratings.

## Problem
We want to answer the question: **Does iterating with cheap models via the evolution pipeline produce better articles than a single call to an expensive model?** To answer this, we need to generate articles using both approaches on the same topics, then compare them in a way that is fair, persistent, and independent of within-run Elo ratings. Currently, the pipeline requires a full article as input (no prompt-based seeding), only supports cheap models in its allowed list, and has no mechanism for cross-run or cross-method comparison.

## Options Considered

### Option A: Extend the Evolution Pipeline Internally
Add 1-shot articles as additional variants injected into evolution runs. Rejected because this would contaminate within-run Elo ratings and make it impossible to isolate the comparison.

### Option B: Article Bank as Separate System (Chosen)
Build a standalone "article bank" concept — a persistent pool of articles grouped by topic/prompt, generated by various methods, with its own Elo comparison system that runs independently from evolution pipeline internals.

### Option C: Manual Comparison Only
Generate articles separately and compare by hand. Rejected because it doesn't scale and lacks the rigor of Elo-based ranking.

## Cost Tracking Strategy

A core requirement is knowing the **total cost to produce each article** in the bank, enabling cost-vs-quality analysis. Cost computation differs by generation method:

### 1-Shot Generation (`generation_method = 'oneshot'`)
- **What counts**: The single LLM call to generate the article + the title generation call
- **Source**: `llmCallTracking` table, filtered by `call_source = 'oneshot_<model>'` and the generation timestamp
- **Computation**: Sum of `estimated_cost_usd` for all calls in that generation session
- **Stored on**: `article_bank_entries.total_cost_usd` at insert time
- **Metadata** (snapshotted at insert time):
  ```jsonc
  {
    "model": "gpt-4.1",
    "generation_time_ms": 3200,
    "prompt_tokens": 480,
    "completion_tokens": 2100,
    // For debugging: which prompt templates and call_source values were used
    "call_source": "oneshot_gpt-4.1",
    "prompt_templates": ["createTitlePrompt", "createExplanationPrompt"],
    // Timestamp window for cross-referencing llmCallTracking rows
    "generation_started_at": "2026-02-01T12:00:00Z",
    "generation_ended_at": "2026-02-01T12:00:03Z"
  }
  ```
- **Debug path**: The `call_source` + timestamp window enables querying `llmCallTracking` for the exact rows (including raw `prompt`, `content`, and `raw_api_response`) if deeper debugging is needed

### Evolution Pipeline Winner (`generation_method = 'evolution_winner'`)
- **What counts**: The **entire run's cost** (`content_evolution_runs.total_cost_usd`) — all agent calls across all iterations. This is the fair measure because the pipeline's value proposition is iterative improvement, and the iteration cost is what we're evaluating.
- **Source**: `content_evolution_runs.total_cost_usd` for the linked run
- **Stored on**: `article_bank_entries.total_cost_usd` copied from run at insert time
- **Metadata** (snapshotted at insert time from `run_summary` + `llmCallTracking`):
  ```jsonc
  {
    // Run execution
    "iterations": 5,
    "duration_seconds": 120,
    "stop_reason": "completed",
    "final_phase": "COMPETITION",
    "seed_model": "deepseek-chat",
    "seed_cost_usd": 0.002,
    // Agent cost breakdown (from llmCallTracking grouped by call_source)
    "agent_cost_breakdown": [
      { "agent": "generation", "calls": 12, "cost_usd": 0.018 },
      { "agent": "calibration", "calls": 24, "cost_usd": 0.009 },
      { "agent": "tournament", "calls": 8, "cost_usd": 0.004 }
    ],
    // Tournament/match structure (from run_summary)
    "match_stats": { "totalMatches": 32, "avgConfidence": 0.85, "decisiveRate": 0.72 },
    "strategy_effectiveness": { "structural_transform": { "count": 3, "avgElo": 1280 }, "crossover": { "count": 2, "avgElo": 1310 } },
    "top_variants_count": 8,
    "winning_strategy": "crossover",
    // Meta-feedback (from run_summary.metaFeedback)
    "meta_feedback": {
      "successfulStrategies": ["crossover", "mutate_clarity"],
      "recurringWeaknesses": ["verbose introductions"]
    }
  }
  ```
- **Seed cost**: If the evolution run was prompt-seeded (Phase 2), the seed article generation cost is included in the run total via `llmCallTracking`
- **Rationale for snapshotting**: Metadata is copied at insert time rather than queried on-demand because evolution runs or their `llmCallTracking` rows may be cleaned up over time, but bank entries should remain self-contained with full provenance

### Evolution Baseline (`generation_method = 'evolution_baseline'`)
- **What counts**: Only the seed generation cost (same as 1-shot), NOT the evolution run cost. This represents "what the cheap model produced before evolution."
- **Source**: Computed from the seed generation `llmCallTracking` entries
- **Stored on**: `article_bank_entries.total_cost_usd`
- **Metadata**: `{ model, prompt_tokens, completion_tokens }` in JSONB

### Comparison/Judging Cost (Not Attributed to Entries)
- Bank comparisons (`article_bank_comparisons`) have their own cost from LLM judge calls
- This cost is tracked in `llmCallTracking` with `call_source = 'bank_comparison'` but is NOT added to any entry's `total_cost_usd` — it's the cost of evaluation, not generation

### Cost-Efficiency Metric

To systematically compare generation methods, the bank computes a normalized **cost-efficiency score** for each entry:

```
elo_per_dollar = (elo_rating - 1200) / total_cost_usd
```

Where 1200 is the Elo baseline (starting rating). Higher values mean more quality per dollar spent. This metric is stored on `article_bank_elo` as a computed column (updated whenever Elo changes) and displayed in the leaderboard and cross-topic summary.

**Edge cases:**
- `total_cost_usd = 0`: `elo_per_dollar` is NULL (division by zero)
- `elo_rating < 1200`: `elo_per_dollar` is negative, meaning "worse than baseline per dollar spent." This is valid and displayed in the UI with red text for negative values. Negative values are included in cross-topic averages (they pull down a method's aggregate score, which is correct).

### 1-Shot Cost Cap

1-shot generation via `generate-article.ts` enforces a **per-call cost cap of $5.00 USD** (configurable via `--max-cost`). Before calling the LLM, the script estimates cost from the prompt token count and the model's pricing. If the estimated cost exceeds the cap, generation is aborted with a clear error. This prevents runaway spend from malformed prompts or retry loops. The evolution pipeline already has `budget_cap_usd`; this is the 1-shot equivalent.

For cross-topic aggregate analysis, the topic list page shows:
- **Avg Elo by method**: mean Elo across all topics for each `generation_method`
- **Avg cost by method**: mean `total_cost_usd` across all topics for each method
- **Avg cost-efficiency by method**: mean `elo_per_dollar` across all topics
- **Win rate by method**: percentage of topics where each method has the highest Elo

## Provenance & Generation History Drill-Through

A core requirement is the ability to trace any bank entry back to its **complete generation history** — every candidate considered, every phase transition, every match played. The existing DB schema already stores this data; the bank just needs to link to it.

### Data Available per Generation Method

**1-Shot entries** (`generation_method = 'oneshot'`):
- Provenance is fully captured in the `metadata` JSONB (model, tokens, cost, generation time)
- No further drill-through needed — the entire generation is a single LLM call

**Evolution entries** (`generation_method = 'evolution_winner'` or `'evolution_baseline'`):
- `article_bank_entries.evolution_run_id` FK → `content_evolution_runs` — links to the full run
- `article_bank_entries.evolution_variant_id` FK → `content_evolution_variants` — links to the specific winning variant
- From the linked run, the following is available via existing infrastructure:
  - **All candidates**: `content_evolution_variants` table — every variant generated during the run, with `agent_name` (which agent created it), `generation` (iteration number), `elo_score`, `quality_scores` JSONB, `parent_variant_id` (lineage), `match_count`
  - **Phase-by-phase history**: `evolution_checkpoints` table — full serialized `PipelineState` at each iteration, including pool contents, Elo ratings, match history, critiques, similarity matrix, diversity scores, and meta-feedback
  - **Agent activity per iteration**: Checkpoint diffs reveal which agents ran, how many variants they added, and how many matches were played each iteration (already computed by `getEvolutionRunTimelineAction`)
  - **Elo progression**: Checkpoint Elo snapshots show how every variant's rating evolved over time (already visualized by `getEvolutionRunEloHistoryAction`)
  - **Lineage DAG**: `parent_variant_id` chains reconstruct the full family tree — which variants were parents of the winner, which strategies produced the best offspring (already visualized by `getEvolutionRunLineageAction`)
  - **Per-agent cost breakdown**: `llmCallTracking` rows filtered by the run's time window and `call_source LIKE 'evolution_%'` (already computed by `getEvolutionRunBudgetAction`)

### UI Drill-Through Path

The bank UI is designed so debugging is **≤2 clicks from the leaderboard** for any entry type:

**Evolution entries** — two paths, both starting from the leaderboard:
1. **1-click path (source link icon)**: Click the external-link icon in the "source" column → goes directly to `/admin/quality/evolution/run/[runId]`, the existing 5-tab run detail page:
   - **Timeline tab**: per-iteration agent activity, variants added, matches played, phase transitions
   - **Elo tab**: Elo curves for all variants across iterations, showing how the winner emerged
   - **Lineage tab**: DAG visualization of parent→child variant relationships
   - **Budget tab**: per-agent cost breakdown pie chart + cumulative burn curve
   - **Variants tab**: full text of every variant with Elo scores and match counts
2. **1-click path (expand row)**: Click the row → inline accordion expands with snapshotted metadata summary (agents, costs, match stats, strategy effectiveness). From here, "Open Run Detail →" and "Compare →" buttons are 1 more click.
3. **Compare page** (`/admin/quality/evolution/run/[runId]/compare`): side-by-side diff of original text vs winner with quality radar chart — reachable from expanded row in 2 clicks.

**1-shot entries** — simpler provenance:
1. **1-click path (source link icon)**: Opens inline LLM call details showing prompt template, model, tokens, cost, and timestamp window for cross-referencing `llmCallTracking` rows.
2. **1-click path (expand row)**: Same metadata inline, plus article text preview.

**Deep debugging** (rare, for investigating anomalies):
- From expanded evolution row → "Open Run Detail →" → run page → Variants tab → find specific variant → see full text, agent, parent, Elo, quality scores. This is the "all candidates" view.
- From expanded oneshot row → use `call_source` + timestamp to query `llmCallTracking` directly for raw prompt/response (admin DB access).

This means **no new visualization infrastructure is needed for provenance** — the existing run detail page already provides complete drill-through. The bank UI adds a source link column and inline row expansion for fast access.

## User Workflows

### Workflow A: Generate 1-Shot Article → Add to Bank

**Goal**: Generate an article on a topic using a single expensive model call and add it to the article bank for comparison.

#### Via CLI

```bash
# Step 1: Generate article and add to bank in one command
npx tsx scripts/generate-article.ts \
  --prompt "Explain quantum entanglement" \
  --model gpt-4.1 \
  --bank \
  --output quantum_entanglement_gpt41.md

# This does:
# 1. Generates title from prompt (via createTitlePrompt)
# 2. Generates full article (via createExplanationPrompt)
# 3. Validates format
# 4. Writes markdown to --output file
# 5. With --bank: creates or finds topic by prompt, inserts entry with:
#    - generation_method: 'oneshot'
#    - model: 'gpt-4.1'
#    - total_cost_usd: computed from llmCallTracking
#    - content: the generated article
# 6. Prints: topic_id, entry_id, cost

# Generate another article on same topic with different model
npx tsx scripts/generate-article.ts \
  --prompt "Explain quantum entanglement" \
  --model claude-sonnet-4-20250514 \
  --bank

# Topic matching: --bank matches existing topic by exact prompt text.
# Same prompt → same topic → new entry alongside the first one.
```

**Topic matching strategy**: Case-insensitive, trimmed prompt match. The `article_bank_topics` table has a UNIQUE constraint on `LOWER(TRIM(prompt))` (functional index). When `--bank` is passed, the script does an atomic upsert via `INSERT INTO article_bank_topics (prompt, title) VALUES ($1, $2) ON CONFLICT (LOWER(TRIM(prompt))) DO UPDATE SET prompt = EXCLUDED.prompt RETURNING id`. This handles both race conditions (two concurrent `--bank` calls with same prompt) and case normalization ("Explain quantum entanglement" and "explain quantum entanglement" map to the same topic).

#### Via Admin UI (`/admin/quality/article-bank`)

1. Click "Generate New Article" button on the topic list page
2. Dialog with fields:
   - **Prompt** (required): text input, e.g. "Explain quantum entanglement"
   - **Model** (required): dropdown of `AllowedLLMModelType` values
   - **Topic**: auto-populated from prompt, or select existing topic
3. Click "Generate" → shows progress spinner → article preview
4. Click "Add to Bank" → saves entry with computed cost

The admin UI generation calls the same server action that the CLI uses, ensuring consistent behavior.

### Workflow B: Add Evolution Run Winner → Bank

**Goal**: Take the winning variant from a completed evolution run and add it to the article bank for cross-method comparison.

```bash
#    - total_cost_usd: copied from content_evolution_runs.total_cost_usd
#    - evolution_run_id: linked to the run

#    - metadata: { iterations, duration_seconds, stop_reason, seed_model, seed_cost_usd,
#                  agent_cost_breakdown, match_stats, strategy_effectiveness,

# 3. Optionally also adds the baseline (seed article) as a separate entry:
#    - generation_method: 'evolution_baseline'

```

#### Via Admin UI — From Evolution Run Detail Page

The existing run detail page (`/admin/quality/evolution/run/[runId]`) already shows the run's variants, winner, and cost. Integration:

   - Only shown when run status is `'completed'` and has a winner (`is_winner = true` variant)
2. Click → dialog with:
   - **Prompt / Topic**: text input (required — the evolution run may not have a stored prompt, so user provides it)
   - **Include baseline?**: checkbox (default: yes) — also adds the original text as a separate `'evolution_baseline'` entry
   - Shows preview: winner text snippet, Elo, total run cost
3. Click "Add" → creates topic (if new) + bank entries
4. Success toast with link to the topic in article bank

#### Via Admin UI — From Article Bank Page

1. On topic detail page (`/admin/quality/article-bank/[topicId]`), click **"Add from Evolution Run"**

   - Each row: run ID, explanation ID, winner Elo, total cost, date
   - Can search/filter by date range

4. Click "Add" → creates bank entry linked to run + variant

2. **Add Anthropic SDK and client routing (adapter layer)**
   - Install: `npm install @anthropic-ai/sdk`
   - New env var: `ANTHROPIC_API_KEY` — add to `.env.example` with placeholder, document in `docs/docs_overall/getting_started.md`. The key is **optional** — only required when using `claude-*` models. If missing and a `claude-*` model is requested, throw a clear error: `"ANTHROPIC_API_KEY required for Claude models"`.
   - File: `src/lib/services/llms.ts` — add `getAnthropicClient()` lazy initializer (matches existing `getOpenAIClient()` / `getDeepSeekClient()` pattern)
   - **Adapter design**: The Anthropic Messages API is fundamentally different from OpenAI's Completions API. Rather than trying to shoehorn Anthropic into `callOpenAIModel`, add a parallel `callAnthropicModel()` function and a thin router:

     ```typescript
     // New: src/lib/services/llms.ts additions
     import Anthropic from '@anthropic-ai/sdk';

     function getAnthropicClient(): Anthropic {
       const key = process.env.ANTHROPIC_API_KEY;
       if (!key) throw new Error('ANTHROPIC_API_KEY required for Claude models');
       return new Anthropic({ apiKey: key, maxRetries: 3, timeout: 60000 });
     }

     function isAnthropicModel(model: string): boolean {
       return model.startsWith('claude-');
     }

     async function callAnthropicModel(
       prompt: string,
       call_source: string,
       userid: string,
       model: string,
       streaming: boolean,
       setText: ((text: string) => void) | null,
     ): Promise<string> {
       const client = getAnthropicClient();
       let response: string;
       let usage: { input_tokens: number; output_tokens: number };

       if (streaming && setText) {
         // Anthropic streaming: MessageStream with content_block_delta events
         let accumulated = '';
         const stream = client.messages.stream({
           model,
           max_tokens: 8192,
           messages: [{ role: 'user', content: prompt }],
         });
         for await (const event of stream) {
           if (event.type === 'content_block_delta' && event.delta.type === 'text_delta') {
             accumulated += event.delta.text;
             setText(accumulated);
           }
         }
         const finalMessage = await stream.finalMessage();
         response = accumulated;
         usage = finalMessage.usage;
       } else {
         // Non-streaming
         const message = await client.messages.create({
           model,
           max_tokens: 8192,
           messages: [{ role: 'user', content: prompt }],
         });
         response = message.content[0].type === 'text' ? message.content[0].text : '';
         usage = message.usage;
       }

       // Map Anthropic usage to common tracking format
       const promptTokens = usage.input_tokens;
       const completionTokens = usage.output_tokens;
       const cost = calculateLLMCost(model, promptTokens, completionTokens, 0);

       await saveLlmCallTracking({
         userid, prompt, content: response, call_source,
         raw_api_response: JSON.stringify({ provider: 'anthropic', model, usage }),
         model, prompt_tokens: promptTokens, completion_tokens: completionTokens,
         total_tokens: promptTokens + completionTokens, reasoning_tokens: 0,
         finish_reason: 'end_turn', estimated_cost_usd: cost,
       });

       return response;
     }

     // Router: raw function that dispatches to the right provider
     async function callLLMModelRaw(...args): Promise<string> {
       if (isAnthropicModel(args.model)) return callAnthropicModel(...);
       return callOpenAIModel(...);
     }

     // IMPORTANT: Wrap with the existing withLogging pattern to preserve
     // automatic logging/timing for all 25+ call sites.
     // The current export chain is:
     //   callOpenAIModelWithLogging = withLogging(callOpenAIModel, ...)
     //   export { callOpenAIModelWithLogging as callOpenAIModel }
     // New chain:
     const callLLMModelWithLogging = withLogging(callLLMModelRaw, 'callLLMModel', ...);
     export { callLLMModelWithLogging as callLLMModel };
     export { callLLMModelWithLogging as callOpenAIModel }; // backward compat
     ```

   - **Structured output**: Anthropic does not support `zodResponseFormat`. For `claude-*` models with `response_obj`, use plain text completion + `JSON.parse()` + Zod `.parse()` validation (same fallback as DeepSeek's `json_object` approach). The prompt should include "Respond in JSON matching this schema:" instruction.

   - **CLI scripts vs Next.js import chain**: `callAnthropicModel` in `llms.ts` calls `saveLlmCallTracking` which uses `createSupabaseServiceClient` (a Next.js server utility). CLI scripts like `generate-article.ts` and `run-evolution-local.ts` cannot import from `llms.ts` without pulling in Next.js deps — this is why `run-evolution-local.ts` already has its own `createDirectLLMClient`.

     **Solution**: Add an Anthropic branch to `createDirectLLMClient` in `run-evolution-local.ts` (and a shared version in `scripts/lib/llmClient.ts`). This duplicates the core Anthropic call logic but avoids the Next.js import chain. The duplication is minimal (the Anthropic Messages API call is ~15 lines) and matches the existing pattern where `createDirectLLMClient` already duplicates the OpenAI/DeepSeek call from `llms.ts`.

     ```typescript

       } else {
         // OpenAI branch (existing)
       }
       // Use getModelPricing() from llmPricing.ts for cost calculation (NOT hardcoded rates)
     }
     ```

   - **`createDirectLLMClient` pricing fix**: Replace the existing hardcoded `inputRate`/`outputRate` with `getModelPricing(model)` from `src/config/llmPricing.ts` (pure function, no Next.js deps, safe to import from scripts). This ensures correct cost tracking for all providers including Anthropic.

   - **CI safety**: `@anthropic-ai/sdk` does NOT validate the API key on import — it only validates on first API call. This means builds and tests won't break when `ANTHROPIC_API_KEY` is unset.

### Phase 3: Article Bank (Persistent Cross-Method Comparison)

**Goal**: Persistent storage and Elo-based comparison of articles from different generation methods on the same topic.

**Changes:**

1. **New DB tables (migration)**

   Migration file: `supabase/migrations/20260201000001_article_bank.sql` (single migration for all 4 tables — atomic, rolls back as a unit if any part fails)

   - `article_bank_topics`: Groups articles by topic/prompt
     - `id` UUID PK DEFAULT gen_random_uuid()
     - `prompt` TEXT NOT NULL — the generation prompt
     - `title` TEXT — generated or provided title
     - `deleted_at` TIMESTAMP WITH TIME ZONE nullable — soft delete (null = active)
     - `created_at` TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
     - UNIQUE index on `LOWER(TRIM(prompt))` — case-insensitive, trimmed matching for topic grouping (supports ON CONFLICT upsert)
   - `article_bank_entries`: Individual articles in the bank
     - `id` UUID PK DEFAULT gen_random_uuid()
     - `topic_id` UUID NOT NULL FK → article_bank_topics ON DELETE CASCADE
     - `content` TEXT NOT NULL — full article text
     - `generation_method` TEXT NOT NULL CHECK (`generation_method IN ('oneshot', 'evolution_winner', 'evolution_baseline')`)
     - `model` TEXT NOT NULL — model used (e.g., `gpt-4.1`, `deepseek-chat`)
     - `total_cost_usd` NUMERIC(10,6) — total cost to produce this article (6 decimal places matches `llmCallTracking.estimated_cost_usd` precision; evolution runs use NUMERIC(10,4) so the extra precision is unused for evolution entries but consistent for 1-shot entries)
     - `evolution_run_id` UUID nullable FK → content_evolution_runs **ON DELETE SET NULL** (if the run is deleted via explanation CASCADE, the bank entry survives with NULL run link — provenance is preserved in the snapshotted `metadata` JSONB)
     - `evolution_variant_id` UUID nullable FK → content_evolution_variants **ON DELETE SET NULL** (same rationale)
     - `metadata` JSONB NOT NULL DEFAULT '{}' — flexible: iterations, run_summary, generation_time_ms, etc.
     - `deleted_at` TIMESTAMP WITH TIME ZONE nullable — soft delete (null = active)
     - `created_at` TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
   - `article_bank_comparisons`: Cross-article Elo match history
     - `id` UUID PK DEFAULT gen_random_uuid()
     - `topic_id` UUID NOT NULL FK → article_bank_topics ON DELETE CASCADE
     - `entry_a_id` UUID NOT NULL FK → article_bank_entries ON DELETE CASCADE
     - `entry_b_id` UUID NOT NULL FK → article_bank_entries ON DELETE CASCADE
     - `winner_id` UUID nullable FK → article_bank_entries ON DELETE SET NULL (null = draw; SET NULL if winner entry is hard-deleted)
     - `confidence` NUMERIC(3,2) CHECK (confidence >= 0 AND confidence <= 1)
     - `judge_model` TEXT NOT NULL — model used for judging
     - `dimension_scores` JSONB nullable — per-dimension results if structured
     - `created_at` TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
   - `article_bank_elo`: Current Elo ratings per topic
     - `id` UUID PK DEFAULT gen_random_uuid()
     - `topic_id` UUID NOT NULL FK → article_bank_topics ON DELETE CASCADE
     - `entry_id` UUID NOT NULL FK → article_bank_entries ON DELETE CASCADE — UNIQUE constraint on (topic_id, entry_id)
     - `elo_rating` NUMERIC(8,2) NOT NULL DEFAULT 1200 CHECK (elo_rating >= 0 AND elo_rating <= 3000)
     - `elo_per_dollar` NUMERIC(12,2) — computed as `(elo_rating - 1200) / total_cost_usd`, updated on every Elo change. NULL if `total_cost_usd` is 0. Can be negative (see Cost-Efficiency Metric edge cases).
     - `match_count` INT NOT NULL DEFAULT 0
     - `updated_at` TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()

   **RLS Policies**: All 4 tables use the same pattern as the existing evolution tables (no RLS enabled — accessed exclusively via service client from server actions behind `requireAdmin()` auth check). This is consistent with `content_evolution_runs` and `content_evolution_variants` which also have no RLS. Comment in migration: `-- No RLS: admin-only access via service client in server actions (requireAdmin guard)`

   **Rollback**: The single migration file wraps all CREATE TABLE statements. If the migration fails, Supabase CLI rolls back the entire transaction. To manually roll back: `DROP TABLE IF EXISTS article_bank_elo, article_bank_comparisons, article_bank_entries, article_bank_topics CASCADE;` (reverse dependency order)

3. **Article bank service**
   - New file: `src/lib/services/articleBankActions.ts`
   - Follows `ActionResult<T>` pattern, `requireAdmin()`, UUID validation, service client — matching `evolutionVisualizationActions.ts`
   - Actions: `addToBankAction`, `getBankTopicAction`, `getBankEntriesAction`, `getBankEntryDetailAction`, `runBankComparisonAction`, `getBankLeaderboardAction`, `getCrossTopicSummaryAction`, `deleteBankEntryAction`, `deleteBankTopicAction`
   - `addToBankAction`: Atomic topic upsert via `INSERT ... ON CONFLICT (LOWER(TRIM(prompt))) DO UPDATE ... RETURNING id`, then insert entry. Uses a transaction for atomicity.
   - `getBankEntryDetailAction`: Returns full entry metadata including snapshotted agent breakdown, match stats, strategy effectiveness. For evolution entries, also returns the `evolution_run_id` for drill-through linking (may be NULL if run was deleted).
   - `getCrossTopicSummaryAction`: Aggregates across all topics — avg Elo by method, avg cost by method, avg elo_per_dollar by method, win rate by method. Used by the topic list page summary panel.
   - `deleteBankEntryAction`: Soft-deletes the entry (`SET deleted_at = NOW()`). Then **explicitly hard-deletes** the associated `article_bank_comparisons` rows (WHERE `entry_a_id = id OR entry_b_id = id`) and the `article_bank_elo` row (WHERE `entry_id = id`). Note: FK ON DELETE CASCADE only fires on actual row deletion, NOT on soft-delete updates. The explicit hard-deletes are necessary because we soft-delete the entry but want comparisons/Elo gone immediately. All listing queries filter `WHERE deleted_at IS NULL`.
   - `deleteBankTopicAction`: Soft-deletes the topic (`SET deleted_at = NOW()`). Then **explicitly hard-deletes** all comparisons and Elo rows for the topic (WHERE `topic_id = id`), and soft-deletes all entries (SET `deleted_at = NOW()` WHERE `topic_id = id`). This preserves entry text for audit while removing active Elo/comparison data.
   - `runBankComparisonAction`: Uses the extracted `compareWithBiasMitigation()` from `src/lib/evolution/comparison.ts`. Passes a `callLLM` callback that calls `callLLMModel` with the judge model and `call_source = 'bank_comparison'`. Updates `article_bank_elo` ratings and `elo_per_dollar`.

## Testing

### CI Safety: Preventing Accidental Live LLM Calls

The project uses **Jest** (not Vitest) — `jest.config.js`, `ts-jest`, `jest.setup.js`. All CI safety mechanisms use Jest APIs.

**Unit tests** (no API keys in CI):
- `jest.setup.js` sets `OPENAI_API_KEY = 'test-openai-key'` (fake key, API calls would auth-fail)
- Add `ANTHROPIC_API_KEY = 'test-anthropic-key'` to `jest.setup.js` (same pattern — prevents import-time errors, but real API calls fail with auth error)
- Add `@anthropic-ai/sdk` to `jest.config.js` `moduleNameMapper` → manual mock at `src/testing/mocks/@anthropic-ai/sdk.ts` (matches existing pattern for `openai`, `@supabase/supabase-js`, `@pinecone-database/pinecone`)

**Integration/E2E tests** (CI DOES set real `OPENAI_API_KEY` via secrets):
- Integration and E2E CI jobs pass real API keys for tests that need them. The article bank integration tests must explicitly mock LLM calls using `jest.spyOn` on the imported `callLLMModel` function (or `createDirectLLMClient`) to prevent billed calls during bank CRUD tests.
- `scripts/generate-article.ts` checks `process.env.NODE_ENV === 'test'` and refuses to call live APIs without an explicit `--allow-live` flag.
- The `run-bank-comparison.ts` integration test mocks the LLM judge call, not the DB operations (those use real Supabase).

**E2E tests** (Playwright):
- E2E comparison flows use pre-seeded bank entries with pre-computed Elo ratings. The "Run Comparison" E2E test verifies UI behavior (button click, progress indicator, leaderboard re-render) by mocking the server action response, not running actual LLM comparisons.
