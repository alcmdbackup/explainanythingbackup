# LLM Costs Too Low In Dashboard Plan

## Background
Evolution dashboard / cost reporting says ~$3 spent in the past week, but real total is more like $40-60. Figure out why by querying Supabase dev and making sure tests are adequately accounted for. Backfill if necessary.

## Requirements (from GH Issue #1263)
- The dashboard ("evolution docs say $3 spent in past week") under-reports LLM spend; real total is ~$40-60 for the same window.
- Investigate root cause by querying **Supabase dev** (`npm run query:staging`, read-only) — compare reported vs actual spend.
- Make sure **tests are adequately accounted for** — confirm the `is_test` discriminator isn't wrongly excluding real operational spend (or wrongly including test spend in the "real" figure).
- **Backfill if necessary** — repair historical/missing cost data where a valid join key exists.

## Problem (confirmed via dev DB — see research doc)
The `/admin/costs` dashboard reads ALL its spend numbers from `llmCallTracking` (summary card, by-model, by-user via direct query; stacked/by-entity via the `get_llm_spend_buckets` RPC; daily via the `daily_llm_costs` view). But `llmCallTracking` is **incomplete**: evolution LLM calls almost never write a row (only 10 of 21,841 invocations over 30d on dev; 479/517 recent real invocations still missing — the audit-gap persists). The evolution source of truth lives separately in `evolution_agent_invocations.cost_usd` (+ `evolution_metrics`). Result: `/admin/costs` sees ~$0.006/30d of evolution spend versus ~$38/30d actual. Separately, **89% of evolution cost is genuine test-strategy spend** that is also filtered/invisible. The foundational defect: there is **no single complete cost ledger** — any consumer that reads `llmCallTracking` under-reports total spend by the evolution amount, and this will recur for every new cost surface.

## Goal (per user decision)
Fix `/admin/costs` to stop under-reporting AND do **foundational rework** so a canonical, complete cost source captures **all** spend (evolution + non-evolution) — so current and future cost consumers cannot silently under-report. Scope: Dev only for now.

> **Canonical file note:** there is exactly ONE cost-analytics module — `evolution/src/services/costAnalytics.ts` (it backs `/admin/costs` despite the `evolution/` path; `src/lib/services/costAnalytics.ts` does NOT exist). All read paths (`getCostSummaryAction`, `getDailyCostsAction`, `getCostByModelAction`, `getCostByUserAction`, `getSpendByGranularityAction`, `getCostByEntityAction`, `getEvolutionReconciliationAction`, `backfillCostsAction`) live there, with its colocated test `evolution/src/services/costAnalytics.test.ts`.

## Options Considered
- [ ] **Option A: Complete the ledger at the source (write-path fix)**: Make every evolution LLM call reliably write a `llmCallTracking` row (genuinely fail-closed on whatever dev path drops them; resolve Open-Q5 — runner agents not using the `requireTracking`-configured provider). Pros: truest "one ledger". Cons: historical gap stays (unbackfillable); must find why writes drop; double-write risk needs dedup discipline. **Does not fix history → not load-bearing.**
- [ ] **Option B-DB (REJECTED — design unsound): DB-level UNION view/RPC of `llmCallTracking` + `evolution_agent_invocations`.** Rejected after plan-review: `evolution_agent_invocations` has NO `model`/`total_tokens`/`userid`/`call_source`/`is_test` columns and is **per-phase grain** (its `cost_usd` aggregates many LLM calls) vs `llmCallTracking`'s per-call grain — so a column-compatible, dedup-safe UNION is not soundly constructible, and by-model/by-user breakdowns are meaningless at invocation grain.
- [x] **Option B-APP (CHOSEN, revised 2026-06-23): app-level canonical merge in `evolution/src/services/costAnalytics.ts`.** Evolution spend is sourced at **invocation grain** — `SUM(evolution_agent_invocations.cost_usd)` bucketed by `invocation.created_at` — by EXTENDING the existing `getEvolutionReconciliationAction` (`evolution/src/services/costAnalytics.ts:692`), which already SUMs invocation cost as the reconciliation truth. **Why invocation grain, not `getRunCostsWithFallback` (run-grain `evolution_metrics` rollup):** (a) it matches the reconciliation oracle's anchor exactly → exact parity, no window-edge disagreement; (b) it counts in-flight/unfinalized runs' spend immediately (the rollup is written only `at_finalization`, so a run-grain source would re-introduce a smaller under-report for live runs — the very bug we're fixing); (c) `/admin/evolution-dashboard` answers a different question (per-run finalized cost) so a small documented difference between the surfaces is correct, and the existing audit-gap banner already compares these two. Non-evolution spend comes from `llmCallTracking` filtered via `attributeCallSource(call_source).category !== 'evolution'` (the mandated, future-proof dedup predicate — excludes the ~$0.04 of evolution rows in `llmCallTracking`; no FK join, no double count). Merged inside the costAnalytics actions; evolution appears as a single `evolution` category/entity (honest about no per-model/per-user breakdown at invocation grain). Pros: extends existing correct reconciliation code, single time anchor, counts live spend, correct across the historical gap, minimal SQL (one index). Cons: app-level merge logic per action; requires a `created_at` index on `evolution_agent_invocations`.
- [x] **Option C: B-APP now + Option A write-path hardening (CHOSEN)**: app-level canonical merge fixes the dashboard immediately (incl. history); write-path hardening (Phase 3) makes the underlying ledger trend complete going forward. Sequence: B-APP does NOT depend on A.
- [x] **Option D: Test-spend classification (CHOSEN — cross-cutting)**: evolution test/real split via the `evolution_runs → evolution_strategies.is_test_content` join (3-table reach from invocations; at run grain it's `run.strategy_id → strategy.is_test_content`); non-evo via `llmCallTracking.is_test`. NULL/orphan strategy classifies as **real** (conservative — never hide real spend). Test spend becomes a visible line, not a filtered-out bucket.
- [ ] **Option E (rejected): Backfill `llmCallTracking` rows for the gap window** — documented non-backfillable (no per-call token data / join key). Data already exists in `evolution_metrics`/`evolution_agent_invocations`; reconstructing the audit table is not the fix.

### Phase 3: Write-path hardening — the "can't silently under-report" guarantee (Option A, 3 layers)
The foundational ask: evolution writes `llmCallTracking` going forward, and *anything* that calls an LLM without recording spend is caught. Three complementary layers (none alone is sufficient — rationale below). **Note:** the real fail-closed gate is `src/lib/services/llms.ts:301` (`if (options?.requireTracking) throw trackingError;`); line 286 is only a log field. Evolution call sites ALREADY pass `requireTracking:true` (`claimAndExecuteRun.ts:223`, arenaActions, runJudgeEval, runPromptEditorConfig), so evolution is already fail-closed *at the chokepoint* — the live bug is calls not reaching it (Phase 0), not a missing default.

- [ ] **Layer 1 — Keep evolution fail-closed + add a kill-switch (do NOT flip the system-wide default).** Confirm every evolution entrypoint passes `requireTracking:true`. Add an env kill-switch (e.g. `LLM_REQUIRE_TRACKING_DISABLED`) so fail-closed can be reverted without a deploy if it starts failing runs. **Explicitly OUT OF SCOPE:** flipping the main-app default to fail-closed — that fail-closes 16 non-evolution `callLLM` sites (stream-chat, aiSuggestion, tagEvaluation, links, etc.) which intentionally swallow tracking blips (`llms.ts:255-256`); a global flip is a separate, risk-assessed decision, not part of this dev-only project.
- [ ] **Layer 2 — Route the direct-provider script through the tracked chokepoint + tighten the CI guard (DECIDED 2026-06-23: track them).**
  - [ ] Extract `claimAndExecuteRun.ts:197-248`'s inline tracked `llmProvider` into a shared factory `createTrackedEvolutionProvider({ db, defaultModel, maxOutputTokens })` (encapsulates `callLLM(..., { trackingDb: db, requireTracking: true, evolutionInvocationId, onUsage })`); refactor the runner to use it (behavior-preserving). Enabler: `callLLM` is already CLI-safe (`claimAndExecuteRun` imports it from `@/lib/services/llms` and runs under `tsx` on the minicomputer — the script's "avoid Next.js import chain" rationale is stale post-06-21 `trackingDb` injection).
  - [ ] `run-evolution-local.ts` (non-`--mock` path): replace `createDirectLLMProvider` with `createTrackedEvolutionProvider({ db: <script's own supabase client> })` so its calls write `llmCallTracking` (fail-closed). Keep `--mock` bypassing real tracking (mocks → no real spend; runtime signal tags `is_test=true` if any row is written).
  - [ ] Remove `'evolution/scripts/run-evolution-local.ts'` from the `check-llm-call-coverage` ALLOWLIST → any future direct-SDK pipeline provider fails CI. Leave the manual probe/benchmark scripts (`test-qwen3-thinking.ts`, `test-oss20b-thinking.ts`, `test-judge-models-v2.ts`, `benchmark-latency.ts`) allowlisted (one-off, not pipeline spend) with a documented reason; `oneshotGenerator.ts` already self-tracks.
  - [ ] **Decide the deliberate side-effects of routing through `callLLM`:** (a) the local script's spend now engages the global `LLMSpendingGate` (daily/monthly caps + semaphore) — desired (real spend should count/cap) but add a bypass flag if experiments must run ungated; (b) `is_test` classification — a real CLI experiment lands `is_test=false` (operational evolution spend) unless a runtime flag tags it.
- [ ] **Layer 3 — Reconciliation monitor (the durable guarantee; catches the silent-divergence class).** Add `evolution/scripts/checkTrackingReconciliation.ts` that, per recent window, compares evolution spend from the **source of truth** (`evolution_metrics`/`evolution_agent_invocations`, per-run SUM — NOT a naive 1:1 row-existence check, since one invocation = N tracking rows) against the `llmCallTracking` evolution total (reuse the shape of the existing `getEvolutionReconciliationAction`, `evolution/src/services/costAnalytics.ts:692`). Wire as a `schedule:`-cron GitHub workflow modeled on **`evolution-cost-alarm.yml`** (files `[release-health]` issues via `gh issue create` — NOT `evolution-run-health.yml`, which is Slack-only), **staging-only** (honor dev-only scope). Define a threshold (e.g. invocation-total > tracking-total × 1.5, mirroring the dashboard banner). **It is a scheduled job, NOT a required PR status check** — so a legitimately-RED reconciliation files an issue and never blocks CI/merges. **Expected RED until Phase 0/Layer 2 lands** — that is the point: it is the detector that proves the write path is fixed and stays fixed. (The Layer-3 *integration test* is separate and MUST pass deterministically on seeded fixtures — it asserts the reconciliation function correctly flags/clears divergence on controlled data, not that live reconciliation is currently green.)

**Rationale (why 3 layers):** Layer 1 throws only for calls that reach the chokepoint; Layer 2 (static CI) is the only thing that can see a call that bypasses the chokepoint via direct SDK; Layer 3 catches the residual "configured correctly yet no row appears" case that neither a runtime throw nor a static scan detects — exactly the failure mode observed in this investigation.
