# Investigate Baseline Variants Evolution Research

## Problem Statement
Investigate (1) why baseline variants in recent stage runs are consistently near the strongest variants in the pool, (2) why the same baseline variant is being reused every run, (3) how `generateFromSeedArticle` uses the baseline for ranking, and (4) why newly generated variants are often rated systematically below 1200 Elo.

## Requirements (from GH Issue #TBD)
- Investigate why baseline variants in the last few stage runs are consistently near the strongest variants in the pool.
- Investigate why the same baseline variant is being used every run.
- Document how `generateFromSeedArticle` uses the baseline for ranking newly generated variants.
- Explain why ratings for newly generated variants are often systematically low (< 1200).

## High Level Summary

1. **Same baseline reuse is by design (not a bug)**: For prompt-based runs, `resolveContent()` in `evolution/src/lib/pipeline/setup/buildRunContext.ts:123-143` selects the highest-`elo_score` variant in `evolution_variants` where `prompt_id=run.prompt_id AND synced_to_arena=true AND generation_method='seed' AND archived_at IS NULL`. For the heavily-used stage prompt `50514a24-cdf3-40e4-a1c1-922009ebd74d` ("Federal Reserve") there is **exactly one** such seed variant (`id=39d3275f-c898-4cdd-9d4c-ccdea7f02360`, content hash `e30a7bdfb3`), so every new run against that prompt re-uses it verbatim. If no `generation_method='seed'` variant exists for the prompt, the run defers to `CreateSeedArticleAgent` which generates one and (via `sync_to_arena`) persists it as the seed for future runs.
2. **Baseline rates near the top because it's a proven article, starts fresh at mu=25/elo=1200, and beats most transforms**: The reused seed is loaded into the pool with a *fresh* default rating (not its historical arena mu/sigma ‚Äî those are loaded only for non-seed `synced_to_arena=true` entries via `loadArenaEntries()`). Judge models then preferentially pick it over structural/lexical/grounding transformations that introduce aggressive restructuring. Combined with OpenSkill's `beta=0` config (aggressive per-match updates), the baseline quickly rises above new variants.
3. **New variants systematically rate <1200 because they start equal to the baseline then lose most pairwise matches**: Variants generated by `GenerateFromSeedArticleAgent` start at `{elo:1200, uncertainty:400/3 ‚âà 133}`, then `rankNewVariant` runs binary-search ranking against a local snapshot that **includes the baseline**. Losses against the strong baseline plus `beta=0`'s aggressive updates push most transforms below 1200 within a handful of matches.
4. **Duplicate-title bug in seed-article generation**: `createSeedArticle.ts:137` (and `generateSeedArticle.ts:102`) build content as ``# ${title}\n\n${articleContent}``. The `buildArticlePrompt` instructs the LLM "Output ONLY the article content, no title", but many LLMs ignore this and emit their own H1. The persisted seed then contains two H1 headers, e.g. the stored Fed seed starts with `# The Federal Reserve: Structure, Functions, and Economic Role\n\n# The Federal Reserve: Structure, Fu‚Ä¶`. One of the historical stage baselines (run `c4057835-‚Ä¶`) literally contains the text `# Title` because the LLM echoed the prompt's placeholder.
5. **Stage schema lag vs docs**: `evolution/docs/data_model.md` describes `evolution_runs.evolution_explanation_id` and an `evolution_explanations` table, but on stage `evolution_runs` has no such column and `evolution_explanations` is empty (0 rows). Current stage code (`buildRunContext.ts`) still reads `explanation_id`/`prompt_id` directly.

## Why Newly Generated Variants Rate <1200 ‚Äî Mechanism

1. `generateFromSeedArticle.ts` spawns N parallel agents. Each agent generates exactly one variant with a single strategy.
2. The new variant starts at the pool-default rating `{elo:1200, uncertainty:400/3}` (from `rating.ts` / `createRating()`).
3. `rankNewVariant` runs binary-search ranking against a deep-cloned local snapshot of `pool + ratings + matchCounts` frozen at the iteration start. The baseline is in that snapshot with a (mostly) default rating too ‚Äî but judges pick it more often.
4. OpenSkill's `beta=0` (see `evolution/docs/rating_and_comparison.md` "Constants Reference" and research doc `judging_accuracy_20260412.md`) makes every loss drop mu aggressively. Paired with high starting `uncertainty` (sigma=8.333) the first few losses produce large mu decreases, pushing elo below 1200.
5. Agent surface/discard logic: once an agent's local elo < top15Cutoff and we're budget-stressed the variant is **discarded** (`surfaced=false`, not written to DB). But the stage data shows `persisted=true` on elo=1115 variants ‚Äî meaning under normal budget the low-rated variants are written. So the pool naturally ends up with a heavy left tail below 1200.

## How `generateFromSeedArticle` Uses the Baseline

- `buildRunContext.resolveContent()` returns `originalText` (the baseline text) and no `seedPrompt` when a seed arena entry exists.
- The orchestrator passes that `originalText` into each `GenerateFromSeedArticleAgent` as `input.originalText`.
- Each agent calls `buildPromptForStrategy(text, strategy)` ‚Üí `buildEvolutionPrompt(preamble, 'Original Text', text, instructions)` to transform the baseline into a new variant.
- The transformed variant is then fed into `rankNewVariant` against the local snapshot ‚Äî which **includes the baseline** as pool entry `pool[0]` (created by the orchestrator with `strategy='baseline'`, `generation=0`, `parentIds=[]`).
- Ranking uses the opponent-selection formula `entropy(pWin) / uncertainty^UNCERTAINTY_WEIGHT` (continuous version of the old triage quartile logic). Because the baseline has default `uncertainty=133`, it's a **high-information opponent** early, so new variants tend to draw the baseline as an opponent in the first few rounds.

## Key Findings (numbered)

1. Reused baseline for prompt `50514a24` is the single `synced_to_arena=true AND generation_method='seed'` variant, selected by `resolveContent` in `buildRunContext.ts`. This is intentional design, not a bug. If no seed exists, `CreateSeedArticleAgent` generates+persists one.
2. The reused seed has a corrupted title (`# ... \n\n# ...`) because the article LLM ignores the "no title" instruction and the pipeline prepends its own H1 unconditionally.
3. The baseline's *arena* rating (mu=18.75) is far lower than its *per-run* rating (up to mu=37.82). Each run loads the seed with fresh default ratings, not arena-history ratings.
4. `run_summary.baselineRank=1` in 6/7 runs where populated ‚Üí baseline usually wins the run.
5. New variants systematically rate below 1200 because (a) they start equal to the baseline, (b) the baseline beats them in judge comparisons, (c) OpenSkill `beta=0` produces aggressive per-match drops, and (d) low-rated discards are still persisted under normal budget.
6. `generateFromSeedArticle` never *scores against* the baseline directly as a reference ‚Äî it uses the baseline as source text for transformation AND as one of many pool members in the local-snapshot binary-search rank.
7. Stage lacks an `evolution_explanation_id` column on `evolution_runs` and `evolution_explanations` has 0 rows ‚Äî the doc description for those is ahead of the stage schema.

## Open Questions

1. **Is fresh-rating-on-load intentional?** Should the seed's persistent arena rating (e.g. mu=18.75 for Fed) be used when the seed is loaded as baseline, instead of defaulting to mu=25? If the seed's arena elo was 1099, starting at 1200 gives it an artificial boost.
2. **Should we fix the double-title generation?** Simplest fix: strip leading `# ‚Ä¶\n` from `articleContent` before concatenating, OR drop the unconditional prepended title.
3. **Is it desirable for the baseline to win 6/7 runs?** If the goal of the pipeline is evolution-toward-better-variants, the baseline repeatedly winning means either (a) the transformations are genuinely worse, (b) judge bias favors the baseline, or (c) the rating math makes wins insufficient to overcome the baseline's "incumbency". The empirical data doesn't distinguish between these.
4. **Arena seed selection policy**: `resolveContent` picks the *highest-elo* seed. For the Fed prompt the lone seed has arena elo 1099 ‚Äî below most arena entries. If the seed is underperforming, should the pipeline promote a top arena entry to `generation_method='seed'` and retire the old one?
